Description

After reading the requirements, I interpreted and structured them below for my own reference.

Create an "available inventory" modal component.

Requirements

  • Validate the email field client-side. Represent the validation state using a green or red border.

    I can validate the state using either a regex or a function that parses the field. Either way, the more stringent validation will occur on the server. I can likely find a 99% solution via google.

  • Respond to submission with either an error or success state.

    This is the only requirement that has deeper technical implications, and thus should be given some consideration.

    The consideration here is how to handle asynchronicity within a component. My initial thought is to use a promise and resolve it when the API call completes. Another approach is to manage the state of the request outside the component.

    There should be a timeout feature to handle the case where the user loses internet connectivity or simply has too much latency. There may be a standard value backed by research for this number. The duration should account for the fact that internet speeds can vary significantly.

    UX-wise, the validation message can present itself either during typing or after the user attempts to submit the form.

    It's usually better to show the error post-submit, as the form will be in some error state when presented with an initially partial email. This is jarring. It gives the wrong impression where the user is progressing normally.

  • Prevent multiple form submissions.

    This will require some state to track an existing submission.

  • Present a loading indicator during submission.

  • When the modal is closed, the form is reset.

    My initial thought is that this will take care of itself. When a React component is "unmounted," its state is loss. I don't foresee any additional work needed to handle this requirement.

  • On form submission, output the product variant ID and the email address to the browser console.

    I understand this is a placeholder for a real API call. I will model the call as if it were making a real async request. That way, in actual usage, only the implementation would need to be changed.

Constraints

  • Limit dependencies to core React.
  • No component libraries.

Conclusion

Some things were not fully implemented due to time constraints.

  • Complete visual states for UX components (hover, focus states).
  • Visual transitions (fading, translating).
  • Responsive, mobile design. The modal should fill the entire page on a narrow screen.
  • Overriding default styles (heavy black border on focused input elements).
